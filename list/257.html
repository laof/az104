HOTSPOT -

You have an Azure subscription named Subscription1. Subscription1 contains the resources in the following table.
VNet1 is in RG1. VNet2 is in RG2. There is no connectivity between VNet1 and VNet2.
An administrator named Admin1 creates an Azure virtual machine named VM1 in RG1. VM1 uses a disk named Disk1 and connects to VNet1. Admin1 then installs a custom application in VM1.
You need to move the custom application to VNet2. The solution must minimize administrative effort.
Which two actions should you perform? To answer, select the appropriate options in the answer area.
NOTE: Each correct selection is worth one point.
Hot Area:

Suggested Answer:

Correct Actions for Moving a VM to Another VNet

First action: Delete VM1.
Second action: Create a new virtual machine.
Explanation of right options: In Azure, you cannot move a VM or its network interface directly to a different VNet. The supported way is to delete the VM while retaining its disk, then create a new VM in the target VNet (VNet2) and attach the original disk. This minimizes administrative effort and ensures the application is preserved, as discussed earlier.
-----------------
First action: Create a network interface in RG2.
Incorrect because a network interface is permanently associated with its VNet at creation; you cannot attach a NIC from another VNet to an existing VM.
First action: Detach a network interface.
Incorrect because detaching the NIC does not allow you to connect the VM to a different VNet; changing the VNet is not supported after VM creation.
First action: Move a network interface to RG2.
Incorrect because moving a NIC to another resource group does not move it to another VNet; the VNet cannot be changed.
Second action: Attach a network interface.
Incorrect because you cannot attach a NIC from a different VNet to an existing VM; VNets are immutable for NICs and VMs.
Second action: Create a network interface in RG2.
Incorrect because even if you create a new NIC in RG2 (and VNet2), you cannot attach it to the existing VM1 which is bound to VNet1.
Second action: Move VM1 to RG2.
Incorrect because moving a VM between resource groups does not change its VNet; the VM will still be in the original VNet1.
