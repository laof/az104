1. VM1 is in the same availability set as VM2: The statement only confirms VM2 is in an availability set, not VM1. The configuration and images do not mention VM1's availability set status. Even if LB1 uses VM2 as a backend, it does not mean the backend pool is defined by an availability set; Azure Load Balancer backend pools can contain individual VMs, those in availability sets, or scale sets.
2. If Probe1.htm is present on VM1 and VM2, LB1 will balance TCP port 80 between VM1 and VM2: With a functional health probe (Probe1.htm) and a rule for TCP/80, LB1 will balance HTTP traffic between the VMs. If a VM fails the probe, it will not receive traffic. The backend pool does not have to be an availability set; it can be individual VMs.
3. If you delete Rule1, LB1 will balance all the requests between VM1 and VM2 for all the ports: Azure Load Balancer requires at least one rule to map frontend IP/port/protocol to the backend pool. Deleting Rule1 means no load balancing for any port; LB1 does not balance all ports by default.
Additional Points:
The phrase "in the same availability set as VM2" confirms only VM2 is in an availability set, not VM1. Using a VM in an availability set as a backend does not mean the backend pool is defined by an availability set. Backend pools can include standalone VMs or VMs in sets or scale sets.
Without rules, no requests are distributed, regardless of backend pool configuration.
