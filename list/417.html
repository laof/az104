HOTSPOT -

You have an Azure virtual machine named VM1 that connects to a virtual network named VNet1. VM1 has the following configurations:
✑ Subnet: 10.0.0.0/24
✑ Availability set: AVSet
✑ Network security group (NSG): None
✑ Private IP address: 10.0.0.4 (dynamic)
✑ Public IP address: 40.90.219.6 (dynamic)
You deploy a standard, Internet-facing load balancer named slb1.
You need to configure slb1 to allow connectivity to VM1.
Which changes should you apply to VM1 as you configure slb1? To answer, select the appropriate options in the answer area.
NOTE: Each correct selection is worth one point.
Hot Area:

Suggested Answer:
Change the private IP address of VM1 to static
— Standard Load Balancer requires backend VMs to have a static private IP to ensure consistent backend pool association; dynamic IPs are not supported for standard SKU backend pools because a dynamic IP can change and break the pool association.
Create and configure an NSG
— As shown in Image 2, if you do not have a Network Security Group (NSG) applied to the subnet or NIC of your VM, no inbound traffic is allowed by default with a Standard Load Balancer; you must create and configure an NSG to explicitly allow the required traffic (such as load balancer health probes and application ports) to reach VM1.
-----------------
Before you create a backend pool on slb1, you must: Create and assign an NSG to VM1 — This is not required for backend pool association; NSGs are for security and traffic filtering, not for pool membership.
Before you create a backend pool on slb1, you must: Remove the public IP address from VM1 — While public IPs should not be on backend VMs for Standard Load Balancer, this is not required before creating the backend pool; it's a best practice but not a technical prerequisite for pool creation.
Before you can connect to VM1 from slb1, you must: Remove the public IP address from VM1 — Although backend VMs for Standard SKU should not have public IPs, the critical requirement for connectivity is having an NSG with the appropriate allow rules; removing the public IP is a best practice but not the key step to enable connectivity.
Before you can connect to VM1 from slb1, you must: Change the private IP address of VM1 to static — This is required before adding to the backend pool, but not specifically for connectivity from the load balancer.
Additional:
Standard vs Basic Load Balancer:
Standard Load Balancer requires static private IPs, does not allow public IPs on backend VMs, and denies all inbound traffic by default unless explicitly permitted by an NSG; Basic Load Balancer allows dynamic IPs, public IPs on backend VMs, and permits inbound traffic unless explicitly denied by NSG.
