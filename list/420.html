Question : 47 Topic: 5

HOTSPOT -

You plan to use Azure Network Watcher to perform the following tasks:
✑ Task1: Identify a security rule that prevents a network packet from reaching an Azure virtual machine.
✑ Task2: Validate outbound connectivity from an Azure virtual machine to an external host.
Which feature should you use for each task? To answer, select the appropriate options in the answer area.
NOTE: Each correct selection is worth one point.
Hot Area:

Suggested Answer:
Task1: IP flow verify — IP flow verify is designed to identify whether a specific network packet is allowed or denied by security rules (such as NSG rules) and to show which rule is responsible for blocking or allowing that flow. It is the primary tool for determining why a packet isn't reaching an Azure VM.
Task2: Connection troubleshoot — Connection troubleshoot (previously known as "Connection monitor") validates outbound connectivity from an Azure VM to an external host by testing and monitoring the connection, which is exactly the requirement for Task2.
It tests the connection at a point in time, rather than monitoring it over time.
-----------------
Task1: Next hop — Incorrect: Next hop is used to determine the next hop in the network route from the VM, not to check security rules preventing packet delivery.
Task1: Packet capture — Incorrect: Packet capture is used to sniff network traffic for diagnostics, not for identifying security rule blocks.
Task1: Security group view — Incorrect: Security group view shows all configured NSG rules but does not simulate network flows or indicate which rule is actively blocking traffic.
Task1: Traffic Analytics — Incorrect: Traffic Analytics provides aggregated analysis of network traffic but not detailed security rule-level diagnostics.
Task2: IP flow verify — Incorrect: IP flow verify only checks if a single packet flow is allowed or denied based on NSG rules; it doesn't validate end-to-end connectivity or diagnose issues outside NSG scope.
Task2: Next hop — Incorrect: Next hop only determines network route next hop, not actual connectivity to external hosts.
Task2: NSG flow logs — Incorrect: NSG flow logs provide logging of allowed/denied flows but do not actively validate or troubleshoot connectivity.
Task2: Traffic Analytics — Incorrect: Traffic Analytics is for traffic flow analysis at a high level, not for validating outbound connectivity from a specific VM to an external host.
