You have an Azure subscription that contains two virtual machines as shown in the following table.

You perform a reverse DNS lookup for 10.0.0.4 from VM2.
Which FQDN will be returned?

  • A..vm1.core.windows.net
  • B..vm1.azure.com
  • C..vm1.westeurope.cloudapp.azure.com
  • CorrectD..vm1.internal.cloudapp.net
Suggested Answer:
D.. vm1.internal.cloudapp.net
Explanation of right option:
With only private IPs and default Azure DNS, a reverse DNS lookup for 10.0.0.4 from VM2 will return vm1.internal.cloudapp.net.
-----------------
Explanation of the other options:
A.. vm1.core.windows.net — Not used for Azure VM DNS.
B.. vm1.azure.com — Not a valid Azure VM FQDN.
C.. vm1.westeurope.cloudapp.azure.com — Only valid for VMs with a public IP, which is not shown in this scenario.
Additional:
According to Azure DNS behavior:
- For VMs with only private IPs and using default (Azure-provided) DNS, the reverse DNS FQDN returned is [vmname].internal.cloudapp.net.
- The FQDN [vmname].[location].cloudapp.azure.com is only registered if the VM has a public IP, which is not indicated in your scenario.
- If the VNet is linked to a private DNS zone as a registration network, both [vmname].[privatednszonename] and [vmname].internal.cloudapp.net can be returned.
