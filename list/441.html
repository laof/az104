You have an app named App1 that is installed on two Azure virtual machines named VM1 and VM2. Connections to App1 are managed by using an Azure Load
Balancer.
The effective network security configurations for VM2 are shown in the following exhibit.
You discover that connections to App1 from 131.107.100.50 over TCP port 443 fail.
You verify that the Load Balancer rules are configured correctly.
You need to ensure that connections to App1 can be established successfully from 131.107.100.50 over TCP port 443.
Solution: You create an inbound security rule that allows any traffic from the AzureLoadBalancer source and has a cost of 150.
Does this meet the goal?

  • CorrectA..Yes
  • B..No
Suggested Answer:
A.. Yes — The first NSG rule already allows client traffic from 131.107.100.50 on port 443. The problem is that the existing deny rule (priority 200) blocks all other 443 traffic, including the Azure Load Balancer health probe. By adding a new rule (priority 150) that allows traffic from AzureLoadBalancer, you unblock the health probe, allowing the VM to be marked healthy in the backend pool. Once healthy, the load balancer can route allowed client traffic to the VM, since the rule for 131.107.100.50 already permits it. Therefore, this solution meets the goal.
-----------------
Explanation of the other options:
B.. No — This option is incorrect because the new rule for AzureLoadBalancer at priority 150 specifically fixes the health probe issue, which is the root cause of the connection failure. With the health probe working, the VM remains available for load-balanced client connections that are already allowed by the first NSG rule.
