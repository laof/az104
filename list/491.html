You have an Azure subscription that contains 10 virtual machines and the resources shown in the following table.

You need to ensure that Bastion1 can support 100 concurrent SSH users. The solution must minimize administrative effort.
What should you do first?

  • A..Resize the subnet of Bastion1
  • B..Configure host scaling.
  • C..Create a network security group (NSG)
  • CorrectD..Upgrade Bastion1 to the Standard SKU
Suggested Answer:
D..Upgrade Bastion1 to the Standard SKU
-----------------
A..Resize the subnet of Bastion1: Incorrect — While subnet size can limit the number of possible sessions, the Basic SKU itself is the primary restriction for concurrent sessions. Resizing alone does not enable support for 100 SSH sessions on Basic SKU.
B..Configure host scaling: Incorrect — Host scaling is a feature available in the Standard SKU of Azure Bastion, not the Basic SKU. You must first upgrade to Standard before host scaling can be configured.
C..Create a network security group (NSG): Incorrect — NSGs control network traffic, but do not affect the maximum number of concurrent SSH users that Bastion1 can support.
D..Upgrade Bastion1 to the Standard SKU: Correct — Upgrading to the Standard SKU is required to support 100 concurrent SSH users, as the Basic SKU does not provide this capability. This also minimizes administrative effort since Standard SKU enables additional scaling and session features.
Additional:
When you configure Azure Bastion using the Basic SKU, two instances are created. If you use the Standard SKU, you can specify the number of instances. This is called host scaling.
Each instance can support 20 concurrent RDP connections and 40 concurrent SSH connections for medium workloads. Once the concurrent sessions are exceeded, an additional scale unit (instance) is required.
