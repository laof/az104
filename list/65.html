HOTSPOT -

You have an Azure subscription named Subscription1 that has a subscription ID of c276fc76-9cd4-44c9-99a7-4fd71546436e.
You need to create a custom RBAC role named CR1 that meets the following requirements:
✑ Can be assigned only to the resource groups in Subscription1
✑ Prevents the management of the access permissions for the resource groups
✑ Allows the viewing, creating, modifying, and deleting of resources within the resource groups
What should you specify in the assignable scopes and the permission elements of the definition of CR1? To answer, select the appropriate options in the answer area.
NOTE: Each correct selection is worth one point.
Hot Area:

Suggested Answer:

Question 2-25

Correct Answer:
"assignableScopes": ["/subscriptions/c276fc76-9cd4-44c9-99a7-4fd71546436e/resourceGroups"]
"notActions": ["Microsoft.Authorization/*"]
Explanation:
Assignable Scopes: To ensure the custom role CR1 can only be assigned to resource groups within the subscription, the assignable scope must be set to the resourceGroups path of the subscription. This restricts the role from being assigned at the subscription or management group level.
Permissions: The role should allow full management of resources (via "actions": ["*"]) but must prevent access management. To do this, "Microsoft.Authorization/*" must be included in "notActions". This blocks permission-related operations while still allowing resource CRUD operations.
-----------------
"/": Incorrect. This would allow assignment at the root level, which is broader than required.
"/subscriptions/c276fc76-9cd4-44c9-99a7-4fd71546436e": Incorrect. This allows assignment at the subscription level, not limited to resource groups.
"/subscriptions/c276fc76-9cd4-44c9-99a7-4fd71546436e/resourceGroups": Correct. This scope restricts assignment to resource groups only.
"Microsoft.Authorization/*": Correct. This prevents the role from managing access permissions.
"Microsoft.Resources/*" and "Microsoft.Security/*": Incorrect. These would block general resource and security operations, which contradicts the requirement to allow full resource management.
