您的公司有多个部门。每个部门都有许多虚拟机（VM）。

该公司有一个 Azure 订阅，其中包含一个名为 RG1 的资源组。
所有虚拟机都位于 RG1。
您希望将每个虚拟机与其 各自的部门 关联起来。
你应该怎么做？

  • A.. 为每个部门创建 Azure 管理组。
  • B.. 为每个部门创建一个资源组。
  • 正确的 C.. 为虚拟机分配标签。
  • D.. 修改虚拟机的设置。
建议答案：
正确选项：C. 为虚拟机分配标签。
说明： 标签 是一种通过将元数据应用于 Azure 资源来组织资源的方法。通过为虚拟机分配标签，您可以将每个 VM 与其各自的部门关联起来，从而更轻松地根据这些标签管理和过滤资源。
-----------------
其他选项说明：
答：为每个部门创建 Azure 管理组： 管理组用于管理跨多个 Azure 订阅的访问权限、策略和合规性。它们不适用于组织单个订阅内的资源。
B. 为每个部门创建一个资源组： 资源组用于管理和组织 Azure 订阅中的资源。但是，由于所有虚拟机都已位于 RG1 中，因此创建额外的资源组并非将虚拟机与部门关联的最有效方法。
D. 修改虚拟机设置： 修改虚拟机设置无法提供将虚拟机与部门关联起来的结构化方法。使用标签是一种更有效、更有条理的方法。

必须将策略配置为要求全局管理员组的成员在从不受信任的位置连接到 Azure AD 时使用多重身份验证和已加入 Azure AD 的设备。
解决方案：您访问 Azure 门户来更改 Azure AD 条件访问策略的授权控制。
解决方案是否达到目标？

  • Correct 答： 是的
  • B.. 没有
建议答案：
正确选项：A.是
说明：更改Azure AD 条件访问策略的 授权控制 可以实现此目标，因为它允许您为多重身份验证指定所需的条件，并为从不受信任位置连接的全局管理员指定已加入 Azure AD 的设备。此配置可确保正确实施策略。
-----------------
其他选项说明：

额外的：

配置 Azure AD 条件访问策略的过程：
1. 访问 Azure AD 门户： 使用管理凭据登录 Azure AD 门户。
2. 导航到条件访问： 转到安全设置并选择条件访问。
3. 创建新策略： 点击“新策略”开始创建策略。
4. 定义条件： 指定策略适用的条件，例如用户组、应用程序和位置。
5. 设置访问控制： 确定访问控制，例如要求多重身份验证 (MFA) 和设备合规性。
6. 启用并保存： 启用该策略，并点击“创建”保存。
-----------------
授权控制： Azure AD 条件访问策略中的授权控制允许您指定授予访问权限必须满足的要求。这包括需要多重身份验证 (MFA)、兼容设备或特定身份验证方法等条件。配置授权控制时，您可以选择用户必须满足的多个要求才能访问资源。
-----------------
说明： 授权控制对于实施安全措施以及确保只有经过身份验证且合规的用户才能访问敏感资源至关重要。通过设置授权控制，您可以增强组织的安全态势并防止未经授权的访问。

问 5

您计划将 Ubuntu Server 虚拟机部署到公司的 Azure 订阅。

您需要实施自定义部署，其中包括添加特定的受信任根证书颁发机构 (CA)。
您应该使用以下哪项来创建虚拟机？

  • A.. New-AzureRmVm cmdlet。
  • B.. New-AzVM cmdlet。
  • C.. Create-AzVM cmdlet。
  • Correct D.. az vm create 命令。
建议答案：
正确选项：D. az vm create 命令。
说明： az vm create 命令用于在 Azure CLI 中创建虚拟机。它允许自定义配置，包括在部署过程中添加受信任的根证书颁发机构 (CA)。
-----------------
其他选项说明：
答：New-AzureRmVm cmdlet： 此 cmdlet 是旧版 AzureRM 模块的一部分，该模块已被 Az 模块取代。不建议在新部署中使用它。
B. New-AzVM cmdlet： 此 cmdlet 在 Az PowerShell 模块中用于创建虚拟机，但它不提供与 az vm create 命令相同级别的自定义。
C. Create-AzVM cmdlet： 此 cmdlet 在 Az PowerShell 模块中不存在。在 PowerShell 中创建 VM 的正确 cmdlet 是 New-AzVM。

额外的：

使用受信任的根 CA 创建 Ubuntu Server VM 的示例：
要使用 Azure CLI 创建 Ubuntu Server 虚拟机并添加受信任的根证书颁发机构 (CA)，可以使用以下命令：
az vm create --resource-group RG1 --name MyUbuntuVM --image UbuntuLTS --admin-username azureuser --generate-ssh-keys --custom-data cloud-init.txt
在此示例中， cloud-init.txt是一个包含用于添加受信任根 CA 的 cloud-init 配置的文件。cloud -init.txt 的内容 可能如下所示：
#云配置
ca-证书：
值得信赖：
- 内容 : |
-----开始证书-----
MIIDdzCCAl+gAwIBAgIEU1iO/jANBgkqhkiG9w0BAQsFADBoMQswCQYDVQQGEwJV
...
-----结束证书-----
文件名 ：/usr/local/share/ca-certificates/my-ca.crt
-----------------
说明： 此命令使用 UbuntuLTS 映像 在资源组 RG1 中创建名为 MyUbuntuVM的虚拟机。cloud -init.txt 文件用于配置虚拟机以包含受信任的根 CA。

解决方案：您可以使用现有多重身份验证提供程序数据的备份来创建一个新的多重身份验证提供程序。
解决方案是否达到目标？

  • Correct 答： 是的
  • B.. 没有
建议答案：
概括：
创建 MFA 提供程序后无法更改使用模型： 一旦创建 MFA 提供程序，就无法更改使用模型。
-----------------
正确流程：
要将使用模型更改为“按启用用户”模式，您需要删除现有的 MFA 提供程序，并创建一个具有所需使用模型的新提供程序。请按以下步骤操作：

3. 配置 MFA 设置： 为新提供商配置 MFA 设置，以确保所有用户（包括新添加的员工）都必须使用多重身份验证。
4. 保存更改： 保存新配置以应用设置。

您的公司有一个名为 weyland.com 的 Azure Active Directory (Azure AD) 租户，该租户配置为与本地 Active Directory 混合共存
目录域。

您在本地 Active Directory 中创建一个新的用户帐户。现在您需要立即将用户信息复制到 Azure AD。
解决方案：运行 Start-ADSyncSyncCycle -PolicyType Initial PowerShell cmdlet。

  • 答： 是的
  • Correct B.. 没有

说明： Start -ADSyncSyncCycle -PolicyType Initial cmdlet 会触发 本地 Active Directory 和 Azure AD 之间的 完整同步周期。这涵盖所有更改，包括 新创建的用户 。但这可能会花费太长时间。因此，此解决方案 无法满足 将新用户信息立即复制到 Azure AD 的目标。
